# UX_PHILOSOPHY.md

# User Experience Philosophy

Aquard is not a sensor dashboard.

Aquard is a decision dashboard.

The purpose of Aquard is not to display measurements.

The purpose of Aquard is to help users understand the state of their water immediately.

---

# The Three Second Rule

A user should understand the current state of their water within three seconds.

The interface should answer the following questions immediately:

- Is my water healthy?
- Can I safely use it?
- Do I need to take action?

Everything else is secondary.

---

# Progressive Disclosure

Never overwhelm the user.

Always present information in layers.

Layer 1

Overall status.

Examples:

- Ready
- Heating
- Attention
- Unsafe

Layer 2

Essential measurements.

Examples:

- Water temperature
- pH
- ORP

Layer 3

Advanced information.

Examples:

- EC
- TDS
- Calibration
- Sensor diagnostics
- Graphs
- History

Advanced information should never distract from the primary experience.

---

# One Primary Focus

Every screen must have one clear visual focal point.

Users should instinctively know where to look first.

Avoid interfaces where every element competes equally for attention.

---

# Minimal Cognitive Load

Aquard should reduce thinking.

Never force users to interpret numbers when a clear status can be provided.

Prefer:

Water is healthy.

Instead of:

pH = 7.46
ORP = 721

Measurements explain the status.

They should not replace it.

---

# Actions Before Data

If Aquard can determine the next action, show the action.

Examples:

No action required.

Add chlorine.

Reduce pH.

Wait until target temperature is reached.

The recommendation is often more valuable than the measurements.

---

# Progressive Interaction

The interface should stay clean.

Advanced information should expand naturally.

Examples:

- expandable panels
- detail drawers
- additional pages

Do not expose every available option by default.

---

# Consistency

The same state should always look the same.

Ready should always use the same colors.

Warnings should always use the same visual language.

Users should build recognition over time.

---

# Motion

Animations exist to communicate state.

Animations should never exist only because they look attractive.

Good examples:

- expanding panels
- status transitions
- button feedback
- loading indicators

Avoid distracting animations.

---

# Design Language

Aquard should feel:

- premium
- calm
- modern
- elegant
- trustworthy

Never playful.

Never noisy.

Never cluttered.

---

# User First

Every implementation should answer:

Does this make the user's life easier?

If not, reconsider the feature.

---

# Final Principle

Users should never have to understand water chemistry in order to understand Aquard.

Aquard understands the water.

The user understands Aquard.
